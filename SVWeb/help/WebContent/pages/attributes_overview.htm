Service terms

A service encapsulates a set of banking activities performed for customers. Each service describes product functionality by means a set of corresponding parameters. These parameters are referred to as service terms in the system. Thus a service is a group of service terms collected for one system object. Service terms can have either simple or complex data types:

•Simple — values can be numeric (for example, a logical variable), CLOB, character (for example, currency) and date (for example, the start date of a bonus). The processing of a simple service terms is hard coded. Examples of simple service terms include: Closure account status, Account currency.

•Complex — values of complex service terms (refer to objects, such as Cards, Terminals, Accounts, etc.) are fees, cycles, limits. The processing of a complex service term can be configured by the user. Examples of complex service terms include: Merchant maintenance fee, Card life cycle length, Wrong PIN entry limit.

For convenience, service terms can be combined into groups. Such grouping is performed using a special type of service term — Group of attribute type. Values cannot be specified for service terms of this type. The type is only used to combine service terms. Examples of service terms of the Group of attribute type include: Account maintenance, Aging and penalties, Grace period.

A service term is created at a service type level. A scale can be attached when a service term is created. A scale is used when the service term value differs from some parameters (modifiers) used during the creation of the scale. For example, to calculate and apply a fee that depends on the transaction type:

1.Create the scale of the “Product attribute parametrization” type

2.Configure the scale modifier using the Operation type parameter.

3.Assign the new scale to the relevant fee.

For more information about the scale configuration, see Scales and modifiers.

Service terms are also affected by the Definition level parameter, which specifies the level at which the value of the term can be set. A validity period is set for each value.

The following rules are valid for inherited service terms (child product):

•The values of service terms assigned the Product and object definition level can be edited for specific child products.

•Values of the service terms with the Service definition level are set at the service level only and cannot be changed at the product level. If the value must be changed for a child product, then you should create a new product with a service of this type and set the service term value for it.

•Specific values of service terms with the Object definition level can be set at the object level only (via the object page or application).

•If a value is not defined at either the product or object levels, then the service term value is inherited from the parent product (up to the product hierarchy).
