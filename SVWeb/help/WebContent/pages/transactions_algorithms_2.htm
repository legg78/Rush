Posting methods

Posting methods determine how transactions of a particular type are posted, that is, when the relevant accounts (ledger, hold, etc.) are adjusted.

IMPORTANT: A transaction in this context refers to a financial entry affecting an account, rather than a financial transaction such as an ATM withdrawal or funds transfer. A single financial transaction may correspond to multiple entries. For example, a transfer results in one account being debited and another being credited.

Depending on the type of transaction and the affected account, different posting methods are employed. Posting methods are specified in the transaction template for both the debit and credit components of a financial entry.

The posting method is applicable when financial entries are registered and account balances are changed accordingly by SVBO.

Posting method

Description

POSTBUFF — Buffered

The buffered method is applied when transaction amounts cannot be posted immediately to accounts; for example, when multiple sources could potentially request that an account be updated simultaneously.

This method is usually employed when transactions impact the accounts of institutions and agents. The buffered method ensures that multiple transactions are not applied to such accounts simultaneously, which might result inconsistencies or database locks.

The buffered method is also used with customer accounts if they are outside the section in which the processing is performed; for example, if a customer account is not linked to any card and it is used for aggregation.

Section refers to a large group of accounts with the same business purpose (credit, loyalty) and linking level (card, merchant, institution or agent). For example, for credit calculations, which affect only credit accounts, transaction processing impacts the card accounts (which constitute one section) while aggregate amounts affect the institution accounts (which constitutes a second section).

Buffered postings are stored immediately in a dedicated table (buffered) when the transaction is processed. They are then applied to the accounts in order using separate processes.

POSTBULK — Bulk

The bulk method is applied if the posting can be postponed until the end of the current processing session. Transactions are stored in a dedicated buffer in memory and are applied to accounts the processing session completes. Unlike the buffered method, this method is suitable if only if it is guaranteed that the account balances will not be updates by separate processes related to different groups of financial transactions.

The bulk method is usually employed with customer accounts when simultaneous transactions affecting the same account are unlikely; for example, during offline calculation of loyalty card bonuses.

Using this method provides performance gains relative to the immediate method through the use of group operations.

POSTIMDT — Immediate

The immediate method is simplest and most frequently used posting method.

The immediate method results in a transaction being posted at the time that it is processed. This method is used when the account balance should be updated immediately after the posting application.

POSTPEND — Pending

The pending method results in the posting of the transaction being delayed until a defined time. The transaction is temporarily stored in a dedicated table. The relevant date and time must be pre-calculated and are applied to the transaction. The transaction will not be applied to the account until that date and time.

POSTRSRV — Reserved

This reserved method is similar to the pending method. However, using this method ensures that SVBO displays the available balance on the screen taking into account the amount to be posted. The amount is not, however, taken into account during subsequent transaction processing which may occur before the posting takes place.
